# 跨目录便携性验证方案评审报告

> **文档说明**：本报告对 `PORTABILITY-TEST-PLAN.md` 进行评审，评估方案的可行性、覆盖率和风险。

---

## 1. 方案概述

### 1.1 方案目标

验证 AI ThinApp Portable Launchpad Platform 的跨目录便携性（POC 验证项 V8），即：
- 应用目录可复制到任意路径（同一台电脑或不同电脑）
- 复制后，所有文件/注册表操作仍然正确重定向到新目录
- 应用功能正常（注册表读取、文件保存、插件加载等）

### 1.2 方案内容

方案包含以下核心内容：

1. **验证目标**：明确便携性验证的目标（路径无关性、重定向正确性、功能完整性、用户数据便携性）
2. **测试环境**：定义 5 种测试环境（本机、本机不同路径、虚拟机 A/B、U 盘）
3. **测试步骤**：设计 7 个测试步骤（捕获应用、解包、本机不同路径运行、虚拟机 A/B 运行、U 盘运行、修改配置）
4. **测试用例**：定义 8 个测试用例（PT-01 ~ PT-08）
5. **验收标准**：定义通过标准（全部通过/部分通过/少数通过）和优先级划分
6. **自动化脚本**：可选，编写 PowerShell 脚本自动化执行测试

## 2. 方案可行性评估

### 2.1 技术可行性

| 评估项 | 评分 | 说明 |
|--------|------|------|
| Hook 引擎稳定性 | ⭐⭐⭐⭐ | MinHook 成熟稳定，社区活跃，技术风险低 |
| 文件系统重定向 | ⭐⭐⭐⭐ | 技术成熟，已有成功案例（Sandboxie、Turbo.net） |
| 注册表重定向 | ⭐⭐⭐ | 技术可行，但注册表 hive 格式较复杂，需仔细设计 |
| 子进程 Hook 继承 | ⭐⭐⭐ | 技术可行，但多层子进程继承可能不稳定 |
| 跨机器便携性 | ⭐⭐⭐⭐ | 核心设计目标，技术可行（所有依赖都在应用目录中） |
| 系统版本兼容性 | ⭐⭐⭐ | Win10 → Win11 可能存在 API 差异，需充分测试 |
| 移动场景（U 盘） | ⭐⭐⭐⭐ | 技术可行，但外部存储设备性能可能影响体验 |

**综合评分**：⭐⭐⭐⭐（4/5）

**结论**：方案技术可行，核心技术（Hook 引擎、文件系统/注册表重定向）已有成熟方案参考，风险可控。

---

### 2.2 操作复杂度

| 评估项 | 复杂度 | 说明 |
|--------|--------|------|
| 测试环境准备 | 中 | 需要准备虚拟机（Win10 + Win11），若 License 受限可能影响进度 |
| 应用捕获 | 低 | AppCapture 工具自动化，操作简单 |
| 测试执行 | 中 | 部分测试需手动（跨机器、U 盘），耗时较长 |
| 结果验证 | 中 | 需要结合手动验证和工具监控（Process Monitor、RegEdit） |
| 自动化脚本 | 高 | PowerShell 脚本编写需要一定技能，且部分测试无法自动化（跨机器） |

**综合复杂度**：中等

**结论**：方案操作复杂度中等，主要挑战在于测试环境准备和跨机器测试的手动执行。建议提前准备虚拟机环境，并考虑部分自动化（如本机测试自动化）。

## 3. 风险识别

### 3.1 测试环境准备风险

| 风险 | 影响 | 概率 | 缓解措施 | 负责人 |
|------|------|------|----------|--------|
| 虚拟机 License 过期 | 测试环境不可用，阻塞测试进度 | 中 | 提前申请 License / 使用免费虚拟机（如 VirtualBox + Win10 评估版） | QA |
| 测试应用下载链接失效 | 无法捕获应用，阻塞测试进度 | 低 | 提前下载安装包，保存到本地 | QA |
| 本机环境损坏 | 需要重新搭建环境，浪费时间 | 低 | 使用系统还原点 / 虚拟机快照 | Dev |

**建议**：
- 提前准备虚拟机环境（最晚 Week 3 开始前完成）
- 提前下载测试应用安装包（保存到本地）
- 本机环境创建系统还原点（防止意外损坏）

- 准备备用方案（如 Hook 引擎失效时，降级到无 Hook 模式）

| 风险 | 影响 | 概率 | 缓解措施 | 负责人 |
|------|------|------|----------|--------|
| Hook 引擎在跨机器环境失效 | 测试结果不准确，误判便携性 | 中 | 在跨机器环境重新安装 Hook DLL；准备日志收集工具 | Dev |
| 注册表 hive 格式不兼容（Win10 → Win11） | 注册表重定向失败 | 低 | 使用通用格式（JSON）；Win11 特定配置单独处理 | Dev |
| 文件重定向性能损耗过高（外部存储） | 用户体验差，便携性大打折扣 | 低 | POC 不优化性能；若损耗过高，MVP 阶段优化 | Dev |

**建议**：
- Hook 引擎在跨机器环境失效时，准备日志收集工具（Process Monitor、API Monitor）
- 注册表 hive 格式使用通用格式（JSON），避免平台依赖
- 性能损耗问题延后到 MVP 阶段优化

## 4. 改进建议

### 4.1 自动化

#### 4.1.1 本机测试自动化

**现状**：PT-01（本机不同路径）需手动执行（复制目录、启动应用、验证）

**建议**：编写 PowerShell 脚本自动化执行 PT-01，包括：
1. 自动复制应用目录到不同路径
2. 自动启动应用
3. 自动检查进程是否运行
4. 自动验证文件重定向（检查 VFS 目录）
5. 自动生成测试报告

**预期收益**：
- 节省手动测试时间（从 30 分钟降到 5 分钟）
- 提高测试一致性（避免人为错误）
- 便于 CI/CD 集成（每日自动测试）

#### 4.1.2 注册表重定向自动化验证

**现状**：PT-08（注册表重定向）需手动验证（使用 RegEdit 加载 hive，检查键值）

**建议**：编写 PowerShell 脚本自动化验证注册表重定向，包括：
1. 使用 `reg load` 加载虚拟 hive
2. 使用 `Get-ItemProperty` 读取注册表键值
3. 对比虚拟 hive 和真实注册表
4. 输出验证报告

**预期收益**：
- 节省手动验证时间（从 20 分钟降到 5 分钟）
- 提高验证准确性（避免人为遗漏）

#### 4.1.3 跨机器测试半自动化

**现状**：PT-02 ~ PT-04 需完全手动（复制目录、启动应用、验证）

**建议**：准备虚拟机快照 + 自动化脚本，包括：
1. 虚拟机快照（干净系统，已安装运行时）
2. 共享文件夹（自动复制应用目录）
3. 虚拟机内自动化脚本（自动启动应用、验证功能、生成报告）
4. 宿主机脚本（自动还原快照、启动虚拟机、收集报告）

**预期收益**：
- 节省手动测试时间（从 1 小时降到 20 分钟）
- 提高测试一致性（避免人为错误）

### 4.2 测试工具开发

#### 4.2.1 VFS 目录对比工具

**需求**：对比复制前后 VFS 目录的差异（文件新增/修改/删除）

**功能设计**：
1. 扫描 VFS 目录，生成文件列表（路径、大小、修改时间、哈希）
2. 对比复制前后文件列表，输出差异报告
3. 支持导出为 JSON/HTML

**技术选型**：
- Python（跨平台，库丰富）
- 使用 `hashlib` 计算文件哈希
- 使用 `jinja2` 生成 HTML 报告

**预期收益**：
- 快速定位文件重定向问题
- 便于回归测试（对比历史报告）

#### 4.2.2 注册表 hive 对比工具

**需求**：对比复制前后注册表 hive 的差异（键值新增/修改/删除）

**功能设计**：
1. 加载虚拟 hive
2. 导出为 JSON
3. 对比复制前后 JSON，输出差异报告

**技术选型**：
- Python（跨平台，库丰富）
- 使用 `winreg` 操作注册表
- 使用 `json` 序列化

**预期收益**：
- 快速定位注册表重定向问题
- 便于回归测试（对比历史报告）
