## 2. 关键技术方案摘要

### 2.1 VFS缓存方案
- **核心思路**：通过内存缓存减少文件重定向的路径计算开销
- **数据结构**：哈希表（O(1)查找）+ 双向链表（LRU排序）
- **关键策略**：LRU淘汰、TTL过期、文件修改检测失效
- **性能目标**：缓存命中率≥80%，重定向延迟<1ms

### 2.2 路径重定向优化方案
- **核心思路**：通过Trie树将路径匹配复杂度从O(n*m)降为O(k)
- **混合方案**：Trie树存储规则 + 缓存高频路径结果
- **性能目标**：路径重定向延迟<0.1ms

### 2.3 Hook覆盖检测增强方案
- **核心思路**：通过多检测点（定时+事件驱动）提高检测实时性
- **自动重装**：指数退避重试（1s→2s→4s→8s，最多5次）
- **性能目标**：检测延迟<1s，重装成功率≥95%

### 2.4 子进程继承完善方案
- **核心思路**：通过`CreateRemoteThreadEx`支持多层继承和WOW64场景
- **异常处理**：注入失败仅记录日志，不崩溃父进程
- **性能目标**：5层继承支持，注入成功率≥99%（非WOW64）

## 4. 风险提示

### 4.1 高风险项
| 风险描述 | 影响 | 缓释措施 |
|----------|------|----------|
| VFS缓存锁管理复杂，可能导致死锁 | 高 | 使用读写锁（SRWLOCK），读不阻塞、写阻塞；进行压力测试 |
| 多层子进程继承稳定性问题 | 高 | 限制最大继承层数为5层；增加递归终止条件；进行5层继承压力测试 |
| WOW64场景注入失败率高 | 中 | 增加32/64位DLL适配；进行WOW64场景专项测试 |

### 4.2 中风险项
| 风险描述 | 影响 | 缓释措施 |
|----------|------|----------|
| Trie树内存占用过高 | 中 | 限制Trie树节点数量；定期清理不常用节点 |
| Hook覆盖检测误报率高 | 低 | 增加检测次数阈值（连续2次检测失败才触发重装） |
| 性能监控开销大 | 低 | 生产环境可关闭监控；采样监控（如每100次操作记录一次） |

### 4.3  blocker应对
若存在技术难点（如高性能并发锁），立即记录到`docs/MVP-BLOCKERS.md`并上报PM，申请技术支持或调整需求。
