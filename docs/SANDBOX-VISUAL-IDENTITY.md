# 沙箱视觉标识方案

## 1. 方案概述
沙箱视觉标识的核心目标是让用户能**直观、无学习成本**地区分“沙箱内运行的应用”和“真实环境运行的应用”，从而感知应用的隔离状态，理解“便携化”和“沙箱”的核心价值。

---

## 2. 备选方案分析

### 2.1 方案A：窗口边框着色
#### 实现方式
- 沙箱内运行的应用窗口边框显示为**绿色**，真实环境运行的应用窗口边框保持**默认颜色**（如Windows默认的浅灰色）。
- 通过Hook应用的窗口过程（`SetWindowLongPtr`修改窗口样式，或子类化窗口过程），在窗口绘制时添加绿色边框。

#### 优点
- **直观性强**：用户一眼就能通过窗口边框颜色判断应用是否在沙箱内运行。
- **覆盖全面**：只要应用有窗口，就能显示标识，无论应用是否在任务栏固定。

#### 缺点
- **兼容性问题**：部分应用（如游戏全屏应用、自定义绘制窗口的应用）可能无法正确显示边框，甚至导致窗口绘制异常。
- **视觉干扰**：部分用户可能觉得绿色边框影响美观。

#### 技术实现难度（参考架构L1-L6）
- **难度层级**：L3（用户态Hook层）
- **实现依赖**：需要Hook窗口相关的API（如`CreateWindowEx`、`SetWindowPos`），或子类化窗口过程，修改窗口的绘制逻辑。
- **可行性**：POC阶段可实现，架构中的Hook引擎（基于MinHook）支持用户态API Hook，技术可行。

---

### 2.2 方案B：窗口水印叠加
#### 实现方式
- 在沙箱内运行的应用窗口右上角（或右下角）叠加**半透明的“🔒 沙箱”角标**（或半透明文字水印）。
- 通过注入DLL到应用进程，在窗口的`WM_PAINT`消息中绘制水印，或使用分层窗口（`WS_EX_LAYERED`）叠加在应用窗口上方。

#### 优点
- **不影响窗口本身**：水印是叠加层，不会修改应用窗口的原有样式，兼容性较好。
- **标识明确**：角标直接显示“沙箱”字样，用户无需学习颜色含义。

#### 缺点
- **易被覆盖**：部分应用（如视频播放器、游戏）会全屏绘制，水印可能被覆盖，无法显示。
- **视觉干扰**：半透明水印可能影响用户对应用内容的查看，尤其是小窗口应用。

#### 技术实现难度（参考架构L1-L6）
- **难度层级**：L3（用户态注入层）
- **实现依赖**：需要注入DLL到应用进程，拦截窗口绘制消息（`WM_PAINT`），并在指定位置绘制水印。
- **可行性**：POC阶段可实现，DLL注入技术成熟，架构支持进程注入（子进程继承Hook已规划）。

---

### 2.3 方案C：任务栏标识
#### 实现方式
- 沙箱内运行的应用在任务栏的图标右上角添加**绿色角标（如✅或小绿点）**，真实环境运行的应用任务栏图标无角标（或红色角标⚠️）。
- 通过操作Windows任务栏的Toolbar控件（`Shell_TrayWnd`下的`ToolbarWindow32`），修改应用图标的叠加图像（Overlay Icon）。

#### 优点
- **符合用户习惯**：Windows任务栏本身支持图标角标（如UWP应用的更新角标、微信的消息角标），用户容易理解。
- **视觉干扰小**：角标体积小，不影响用户对应用内容的查看。
- **聚焦运行中的应用**：用户最关注任务栏中运行的应用，标识出现在最显眼的位置。

#### 缺点
- **依赖任务栏固定**：如果应用没有固定在任务栏（仅临时运行），可能无法显示角标（取决于Windows任务栏的图标管理机制）。
- **实现复杂度高**：操作任务栏Toolbar需要深入Windows Shell编程，不同Windows版本（Win10/Win11）的任务栏结构可能不同，兼容性问题较大。

#### 技术实现难度（参考架构L1-L6）
- **难度层级**：L4（Shell编程层）
- **实现依赖**：需要调用Windows Shell API（如`ITaskbarList3::SetOverlayIcon`）来设置任务栏图标的叠加图像，需要进程有相应的权限。
- **可行性**：MVP阶段可实现，Windows提供官方API支持任务栏图标叠加，技术成熟。

---

### 2.4 方案D：以上组合
#### 实现方式
- 同时实现方案A（窗口边框着色）+ 方案C（任务栏标识），或加入方案B（窗口水印叠加）。
- 例如：沙箱内应用的窗口边框为绿色，同时任务栏图标有绿色角标。

#### 优点
- **双重/三重提示**：用户通过多个视觉通道感知沙箱状态，几乎不会遗漏。
- **容错性强**：如果某一种标识因兼容性问题无法显示，其他标识仍能发挥作用。

#### 缺点
- **实现复杂度高**：需要同时实现多种标识方案，开发工作量大。
- **性能影响**：多种标识同时绘制可能增加系统开销，尤其是低配置电脑。
- **视觉干扰大**：多种标识同时显示可能让用户觉得界面杂乱。

#### 技术实现难度（参考架构L1-L6）
- **难度层级**：L4-L5（组合层）
- **实现依赖**：需要同时实现窗口Hook、Shell编程等多种技术，兼容性测试工作量大。
- **可行性**：MVP阶段可部分实现（如A+C），V2阶段再完善。

---

## 3. 推荐方案及理由

### 3.1 推荐方案
**方案C（任务栏标识）+ 方案A（窗口边框着色）的组合方案**，优先实现任务栏标识，再补充窗口边框着色。

### 3.2 推荐理由
1. **符合用户习惯**：任务栏是用户管理运行中应用的核心入口，用户对任务栏图标的角标提示已有认知（如微信、QQ的消息角标），学习成本极低。
2. **视觉干扰小**：任务栏角标体积小，窗口边框着色仅添加颜色，不会影响用户对应用内容的查看。
3. **覆盖大部分场景**：
   - 对于常驻任务栏的应用（如浏览器、办公软件），任务栏角标能持续提示沙箱状态；
   - 对于临时运行的应用，窗口边框着色能提供直观提示。
4. **技术可行性高**：两种方案的技术难度均在项目POC/MVP阶段的可实现范围内（L3-L4级），架构中的Hook引擎和进程注入能力已规划，能支撑实现。

### 3.3 替代方案
如果任务栏标识的实现进度滞后，可先实现**方案A（窗口边框着色）**作为临时方案，待任务栏标识完成后补充，确保POC阶段有可用的视觉标识。

## 4. 技术实现难度评估（参考架构L1-L6）
假设架构中的L1-L6层级定义如下（基于项目架构文档推断）：
| 层级 | 描述 | 示例 |
|------|------|------|
| L1 | 用户态简单配置/无Hook | 配置文件修改、注册表设置 |
| L2 | 用户态普通API调用 | 文件操作、进程创建 |
| L3 | 用户态API Hook/注入 | MinHook Hook、DLL注入 |
| L4 | Shell/系统级用户态编程 | 任务栏操作、窗口子类化 |
| L5 | 内核驱动级修改 | Minifilter驱动、内核Hook |
| L6 | 硬件/固件级修改 | BIOS修改、硬件虚拟化 |

#### 各方案的技术难度对应
| 方案 | 技术难度层级 | 实现可行性（POC/MVP） |
|------|--------------|------------------------|
| 方案A（窗口边框着色） | L3 | 高（POC阶段可实现） |
| 方案B（窗口水印叠加） | L3 | 高（POC阶段可实现） |
| 方案C（任务栏标识） | L4 | 中高（MVP阶段可实现） |
| 方案D（组合） | L4-L5 | 中（V2阶段可实现） |

#### 项目阶段匹配建议
- **POC阶段**：实现方案A（窗口边框着色），验证视觉标识的可行性，收集用户反馈。
- **MVP阶段**：实现方案C（任务栏标识）+ 方案A（窗口边框着色）的组合方案，提供完整的视觉标识体验。
- **V2阶段**：优化方案C的兼容性（支持Win10/Win11不同版本），可选加入方案B（窗口水印叠加）作为辅助。

## 5. 用户测试计划
为了验证推荐方案的有效性，计划在MVP阶段进行用户测试：
1. **测试对象**：5-10名目标用户（Windows个人用户，无沙箱使用经验）。
2. **测试任务**：启动沙箱内应用→判断应用是否在沙箱内运行→解释判断依据。
3. **评估指标**：
   - 识别准确率：用户能正确识别沙箱内应用的比例（目标≥90%）；
   - 学习成本：用户理解标识含义的平均时间（目标≤30秒）；
   - 视觉干扰：用户对标识的视觉干扰评分（1-5分，目标≤2分）。
4. **迭代优化**：根据测试反馈调整标识的颜色、位置、样式（如将绿色边框改为蓝色，或将角标位置调整到左上角）。

---

## 6. 方案总结
推荐采用“任务栏标识+窗口边框着色”的组合方案，该方案符合用户习惯、视觉干扰小、技术可行，能有效帮助用户区分沙箱内和沙箱外的应用，感知便携化和沙箱的核心价值。技术实现难度在POC/MVP阶段的可承受范围内，建议按阶段逐步落地。
