# 沙箱目录结构可视化方案

## 1. 目标

让用户（即使是小白用户）能够直观理解"应用的所有文件都在一个目录里"这个概念，建立"便携化 = 可复制目录即迁移"的心智模型。

### 1.1 核心设计目标

- **降低认知门槛**：让非技术用户也能理解沙箱隔离的概念
- **可视化文件组织**：直观展示应用文件在 `{AppDir}\VFS\` 目录中的组织结构
- **建立便携化认知**：通过可视化让用户理解"复制目录 = 迁移应用"
- **增强信任感**：让用户看到应用确实没有在真实系统中留下垃圾文件

### 1.2 用户体验目标

- 用户在 30 秒内能找到沙箱内的某个文件
- 用户能理解"所有文件都在 VFS 目录内"的概念（通过可用性测试验证）
- 用户感知到"便携化"的价值（可复制性、隔离性）

---

## 2. 可视化方案对比

### 2.1 方案 A：树状文件浏览器（类似 Windows 资源管理器）

#### 实现方式
- 采用树状目录结构展示 `{AppDir}\VFS\` 目录
- 左侧为可展开/折叠的目录树，右侧为文件列表（名称、大小、修改时间、类型图标）
- 支持文件操作：打开、导出、删除、搜索

#### 优点
- **用户熟悉**：Windows 用户对抗议界面认知度高，学习成本极低
- **操作直观**：符合用户已有的文件管理习惯
- **信息完整**：能展示完整的文件结构细节和文件属性
- **实现难度低**：Electron + Ant Design Tree 组件即可实现

#### 缺点
- **技术感强**：纯文件浏览器界面可能让小白用户觉得"像程序员用的工具"
- **视觉吸引力弱**：缺乏视觉冲击力，难以形成记忆点
- **空间占用大**：需要较大的界面空间来展示目录树和文件列表

#### 技术实现难度
- **难度层级**：L2（用户态普通 API 调用）
- **实现依赖**：Electron + Ant Design Tree 组件（或自绘 TreeView）
- **开发工时估算**：基础版本 3-5 天，完整版本 7-10 天

#### 适用场景
- POC 阶段快速验证用户对"沙箱文件可视化"的接受度
- 需要精确查看文件结构和文件属性的场景

---

### 2.2 方案 B：存储空间可视化（类似 WinDirStat）

#### 实现方式
- 采用矩形树图（Treemap）或环状图（Sunburst）展示 `{AppDir}\VFS\` 目录的存储空间占用
- 矩形大小对应文件/目录大小，颜色编码文件类型
- 支持交互：点击矩形下钻到子目录，鼠标悬停显示文件/目录名称和大小

#### 优点
- **视觉吸引力强**：图形化展示直观且美观，容易吸引用户注意
- **空间占用一目了然**：用户能快速了解哪些文件/目录占用了最多空间
- **现代感强**：符合现代应用的设计趋势（如 Storage Sense、DaisyDisk）
- **差异化竞争**：与竞品（Sandboxie）形成明显差异

#### 缺点
- **无法展示文件结构细节**：只能看到空间占用，无法查看具体的文件列表和属性
- **学习成本**：部分用户可能不熟悉 Treemap 的可视化方式
- **不适合精确操作**：无法方便地进行文件级操作（如导出某个特定文件）

#### 技术实现难度
- **难度层级**：L3（需要遍历文件计算大小）
- **实现依赖**：HTML5 Canvas + JavaScript（或 D3.js）
- **开发工时估算**：基础版本 5-7 天，完整版本 10-14 天

#### 适用场景
- MVP 阶段增强产品视觉吸引力
- 需要展示沙箱空间占用的场景（如存储空间紧张时提示用户清理缓存）

---

### 2.3 方案 C：对比视图（Before / After）

#### 实现方式
- 采用分屏对比展示"真实系统污染" vs "沙箱纯净"
- 左侧展示"未使用沙箱时"应用对真实系统的修改（注册表、系统目录写入等）
- 右侧展示"使用沙箱后"所有修改都被限制在 `{AppDir}\VFS\` 目录内

#### 优点
- **强烈对比**：直观展示沙箱的价值（保护系统清洁）
- **说服力强**：用户能清晰看到"如果没有沙箱，系统会被污染成什么样"
- **营销价值高**：适合用于产品宣传和功能介绍页面

#### 缺点
- **需要截图真实系统状态**：可能涉及用户隐私（真实系统中安装应用会留下大量痕迹）
- **实现复杂度高**：需要虚拟机快照或系统监控工具来记录"Before"状态
- **静态展示局限**：只能展示某个时间点的状态，无法动态展示文件变化

#### 技术实现难度
- **难度层级**：L3-L4（需要虚拟机快照或系统监控）
- **实现依赖**：虚拟机快照管理 或 系统文件监控工具（如 Process Monitor）
- **开发工时估算**：基础版本 7-10 天，完整版本 14-21 天

#### 适用场景
- 产品官网的功能介绍页面
- 首次使用引导流程中的概念介绍步骤
- 营销材料和演示视频

---

### 2.4 方案 D：时间轴视图（文件变化历史）

#### 实现方式
- 采用时间轴（Timeline）展示应用运行过程中的文件变化历史
- 横向时间轴显示时间点，纵向显示文件变化（新增、修改、删除）
- 支持播放动画：模拟应用运行过程中文件的动态变化

#### 优点
- **动态展示**：能展示应用运行过程中的文件变化，生动有趣
- **调试价值**：对高级用户和开发者有调试价值（查看应用行为）
- **故事化叙事**：适合用于展示"应用生命周期"的故事

#### 缺点
- **需要记录文件操作日志**：Hook 层需要记录所有文件操作，性能开销较大
- **实现难度极高**：需要 Hook 层记录日志 + 前端可视化渲染
- **目标用户局限**：主要吸引技术用户，小白用户可能不关心文件变化细节

#### 技术实现难度
- **难度层级**：L5（Hook 层需要记录日志）
- **实现依赖**：Hook 层记录文件操作日志 + 前端时间轴可视化
- **开发工时估算**：基础版本 14-21 天，完整版本 30+ 天

#### 适用场景
- V2 阶段的高级功能（面向开发者和技术爱好者）
- 应用行为分析和调试工具

---

### 2.5 方案 E：3D 立方体可视化（创新方案）

#### 实现方式
- 采用 3D 立方体展示沙箱目录结构
- 立方体的每个面代表一个顶级目录（如 C 盘、D 盘、注册表等）
- 支持旋转、缩放、点击面进入子目录

#### 优点
- **视觉冲击力强**：3D 视觉效果容易吸引注意力，形成记忆点
- **差异化明显**：市面上几乎没有同类产品使用 3D 可视化
- **科技感强**：符合"AI ThinApp"的科技定位

#### 缺点
- **实现难度极高**：需要 Three.js + 复杂的交互逻辑
- **可能分散注意力**：3D 效果可能让用户忽略核心信息（文件结构）
- **性能开销大**：3D 渲染对 GPU 有一定要求
- **学习成本高**：用户需要学习如何旋转、缩放 3D 立方体

#### 技术实现难度
- **难度层级**：L5-L6（Three.js + 复杂交互）
- **实现依赖**：Three.js（3D 渲染库）+ 复杂的交互逻辑
- **开发工时估算**：基础版本 21-30 天，完整版本 40+ 天

#### 适用场景
- V3 阶段的创新功能（作为产品的差异化卖点）
- 产品宣传视频和营销材料

## 3. 推荐方案及理由

### 3.1 推荐方案

**方案 A（树状文件浏览器）+ 方案 B（存储空间可视化）组合方案**

### 3.2 推荐理由

1. **覆盖不同用户需求**：
   - 方案 A 满足需要精确查看文件结构和属性的用户（如高级用户、开发者）
   - 方案 B 满足需要快速了解空间占用的用户（如小白用户、存储空间紧张时）

2. **技术可行性高**：
   - 两种方案的技术难度均在 POC/MVP 阶段的可实现范围内（L2-L3 级）
   - 方案 A 可快速实现 POC 验证，方案 B 可在 MVP 阶段补充

3. **用户体验渐进增强**：
   - POC 阶段实现方案 A（基础文件浏览器），快速验证用户接受度
   - MVP 阶段增加方案 B（存储空间可视化），增强视觉吸引力
   - 体验完善阶段增加"导出沙箱报告"功能（PDF/HTML），提升专业度

4. **与竞品形成差异**：
   - Sandboxie 只有基础的文件夹视图，缺乏存储空间可视化
   - 我们的组合方案能形成明显的产品差异化

### 3.3 分阶段实施计划

| 阶段 | 内容 | 目标 |
|------|------|------|
| **POC 阶段** | 实现方案 A（基础文件浏览器） | 验证用户对"沙箱文件可视化"的接受度 |
| **MVP 阶段** | 增加方案 B（存储空间可视化） | 增强视觉吸引力，展示空间占用 |
| **体验完善阶段** | 增加"导出沙箱报告"功能（PDF/HTML） | 提升专业度，方便用户分享和存档 |

## 4. 技术实现细节

### 4.1 方案 A 实现（树状文件浏览器）

#### 4.1.1 技术选型

| 技术栈 | 选型 | 理由 |
|--------|------|------|
| 前端框架 | Electron | 跨平台，可复用现有 Web 技术栈 |
| UI 组件库 | Ant Design | 提供成熟的 Tree 组件和文件列表组件 |
| 文件读取 | Node.js `fs` 模块 | Electron 主进程可直接访问文件系统 |
| 状态管理 | Redux（可选） | 管理文件树展开状态、选中状态等 |

#### 4.1.2 数据源

- **数据源路径**：`{AppDir}\VFS\` 目录
- **读取方式**：Electron 主进程通过 `fs.readdirSync()` 实时读取目录内容
- **缓存策略**：首次读取后缓存目录结构，文件变化时使用 `chokidar` 监听并增量更新

#### 4.1.3 功能列表

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 展开/折叠目录树 | 点击目录前的箭头展开或折叠子目录 | P0 |
| 搜索文件 | 支持文件名模糊搜索（实时显示匹配结果） | P1 |
| 双击文件打开 | 用系统默认程序打开文件（透传，不在沙箱内运行） | P0 |
| 右键菜单 | 打开文件位置、导出文件、删除文件、查看属性 | P1 |
| 显示文件属性 | 显示文件大小、修改时间、文件类型图标 | P0 |
| 高亮新增/修改文件 | 对比真实系统，高亮显示沙箱中新增或修改的文件 | P2 |

#### 4.1.4 性能优化

| 优化策略 | 说明 | 适用场景 |
|----------|------|----------|
| 懒加载 | 点击展开时才读取子目录内容，避免一次性加载整个目录树 | 文件数量 > 100 |
| 虚拟滚动 | 文件数量 > 1000 时，只渲染可见区域的文件列表项 | 文件数量 > 1000 |
| 后台索引 | 应用启动时在后台线程建立文件索引，加速搜索 | 文件数量 > 5000 |
| 增量更新 | 使用 `chokidar` 监听文件变化，只更新变化的目录节点 | 文件变化频繁时 |

#### 4.1.5 右键菜单设计

```
+-------------------------------------+
| 打开 (Open)                         |  → 用系统默认程序打开文件
| 打开文件位置 (Reveal in Finder)      |  → 跳转到 VFS 目录（在资源管理器中显示）
| 导出 (Export)                       |  → 复制文件出沙箱（选择导出路径）
| 删除 (Delete)                       |  → 从沙箱中删除文件（需确认）
| ----------------------------------- |
| 属性 (Properties)                   |  → 显示文件详细信息（大小、修改时间、路径等）
+-------------------------------------+
```

### 4.2 方案 B 实现（存储空间可视化）

#### 4.2.1 技术选型

| 技术栈 | 选型 | 理由 |
|--------|------|------|
| 可视化库 | D3.js | 成熟的 JavaScript 可视化库，支持 Treemap 和 Sunburst |
| 渲染方式 | HTML5 Canvas | 性能优于 SVG（文件数量 > 1000 时） |
| 数据计算 | Node.js 后台线程 | 遍历文件计算大小，避免阻塞主线程 |
| 颜色方案 | d3-scale-chromatic | 提供科学的颜色映射方案 |

#### 4.2.2 可视化类型

**推荐：矩形树图（Treemap）**

- **优点**：空间利用率高，矩形大小直观对应文件/目录大小
- **缺点**：父目录和子目录的层级关系不如 Sunburst 直观
- **适用场景**：需要快速了解空间占用分布的场景

**备选：环状图（Sunburst）**

- **优点**：层级关系直观（从内到外对应从根目录到子目录）
- **缺点**：空间利用率低（圆心附近的区域很小，难以点击）
- **适用场景**：需要展示目录层级关系的场景

#### 4.2.3 数据源

- **数据源路径**：`{AppDir}\VFS\` 目录
- **数据计算**：遍历目录，计算每个文件和目录的大小
- **数据格式**：
  ```json
  {
    "name": "VFS",
    "children": [
      {
        "name": "C",
        "children": [
          {
            "name": "Windows",
            "size": 1024000000,
            "children": [...]
          }
        ]
      }
    ]
  }
  ```

#### 4.2.4 交互设计

| 交互 | 说明 |
|------|------|
| 点击矩形 | 下钻到子目录（更新 Treemap 数据） |
| 鼠标悬停 | 显示文件/目录名称和大小（Tooltip） |
| 颜色编码 | 文件类型（可执行文件、配置文件、数据文件、缓存文件） |
| 图例 | 文件类型颜色说明（可折叠） |
| 返回上一级 | 面包屑导航或"返回"按钮 |

#### 4.2.5 性能优化

| 优化策略 | 说明 | 适用场景 |
|----------|------|----------|
| 预计算文件大小 | 应用启动时在后台线程遍历文件并计算大小 | 文件数量 > 1000 |
| 分块渲染 | 避免一次性渲染几万个矩形，采用分块渲染或聚合展示 | 文件数量 > 10000 |
| 数据采样 | 文件数量 > 50000 时，只展示前 10000 个最大的文件/目录 | 文件数量极大时 |
| Web Worker | 使用 Web Worker 在后台线程计算 Treemap 布局 | 计算时间较长时 |

#### 5.1.2 交互说明

1. **目录树**：
   - 点击箭头展开/折叠目录
   - 点击目录名称在右侧文件列表中显示该目录的内容
   - 右键点击目录显示上下文菜单（展开所有、折叠所有、刷新）

2. **文件列表**：
   - 双击文件用系统默认程序打开
   - 右键点击文件显示上下文菜单（打开、导出、删除、属性）
   - 支持按名称、大小、修改时间排序（点击列标题）

3. **搜索框**：
   - 输入文件名关键字，实时过滤文件列表
   - 支持模糊搜索（如输入"txt"会匹配所有 `.txt` 文件）

4. **状态栏**：
   - 显示当前目录的总文件数和总大小
   - 显示沙箱已用空间（整个 `{AppDir}` 目录的大小）

| RISK-05 | 用户对 Treemap 可视化方式不熟悉 | 用户无法理解存储空间可视化 | 中 | 在首次使用引导中加入 Treemap 的使用说明；提供传统文件浏览器作为默认视图 | UX |
| RISK-06 | 文件浏览器界面技术感太强，吓跑小白用户 | 小白用户不敢操作 | 高 | 简化界面，隐藏技术细节（如 VFS 路径）；提供"简单模式"和"高级模式"切换 | UX |
| RISK-07 | 右键菜单功能太多，用户找不到需要的功能 | 用户操作效率低下 | 中 | 精简右键菜单，只保留最常用的功能；将不常用的功能移到侧边栏或工具栏 | UX |
