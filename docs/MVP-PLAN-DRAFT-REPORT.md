## 3. 风险提示

### 3.1 POC 评审可能不通过（No-Go）

**风险描述**：若 POC 评审未通过（No-Go），MVP 计划可能作废，项目需要重新评估技术路线。

**可能的原因**（详见 `POC-REVIEW-FRAMEWORK.md`）：
1. **V1 失败**：Hook 引擎无法稳定工作（崩溃/内存泄漏严重）
2. **V2 失败**：文件系统重定向不完整（大量文件泄漏到真实系统）
3. **V5 失败**：杀软持续报毒且无法申请白名单（项目无法分发）

**影响**：
- 项目暂停投入，技术路线调整
- MVP 计划作废，需要重新规划
- 团队士气受损

**缓释措施**：
1. **V1 风险缓释**：充分的单元测试和集成测试；Hook 失败时降级策略（放行而非拦截）；异常捕获 + 日志记录
2. **V2 风险缓释**：POC 阶段不优化性能，仅验证功能；MVP 阶段热路径优化（缓存、异步 IO）
3. **V5 风险缓释**：申请 OV 代码签名证书（~$200/年）；POC 阶段提示用户关闭杀软；开源透明（GitHub 完整源码）

**应对方案**（若 POC 评审 No-Go）：
1. **技术路线不可行**：重新评估方案（如切换到内核 Minifilter 驱动）
2. **杀软误报无法解决**：考虑开源托管 + 用户自编译；或换用其他 Hook 框架（如 Microsoft Detours）
3. **项目终止**：若技术路线和杀软问题都无法解决，考虑终止项目，转向其他方向

### 3.2 MVP 计划评审可能不通过（No-Go）

**风险描述**：若 MVP 计划评审未通过（No-Go），MVP 开发无法启动，需要重新规划。

**可能的原因**：
1. **范围不合理**：MVP 范围过大或过小，不包含的功能不明确
2. **时间计划不可行**：10 周时间不足，任务拆解不详细，依赖关系不清晰
3. **资源不足**：找不到 Windows 底层开发 Lead，或团队成员技能不匹配
4. **技术风险过高**：Hook 引擎性能问题、应用捕获失败率高、杀软误报率高

**影响**：
- MVP 开发无法启动，项目延期
- 需要重新规划 MVP（范围、时间、资源）
- 团队士气受损

**缓释措施**：
1. **范围合理性**：参考 POC 验证结果，明确 MVP 范围（包含/不包含的功能）；使用 MoSCoW 方法（Must have/Should have/Could have/Won't have）排序
2. **时间计划可行性**：任务拆解到天级别；识别关键路径和依赖关系；预留 Buffer（10-20%）
3. **资源充足性**：外部技术顾问（短期过渡）；自研 Inline Hook（降低对 Lead 的依赖）；开源社区求助（发布技术难题，悬赏解决）
4. **技术风险降低**：实现 VFS 缓存（减少文件 IO）；优先捕获 10 款常用应用，逐步扩展；申请 EV 证书，提前提交白名单

### 3.3 MVP 开发阶段可能延期

**风险描述**：MVP 开发阶段可能延期（超过 10 周），导致发布日期推迟。

**可能的原因**：
1. **Hook 引擎性能问题**：文件/注册表 IO 瓶颈，优化困难
2. **应用捕获失败率高**：复杂应用无法捕获，需要大量调试
3. **Launchpad UI 开发进度延迟**：GUI 框架不熟悉，或设计频繁变更
4. **代码签名证书申请延迟**：需 1-2 周，可能更长
5. **团队成员离职或请假**：关键人员离开，导致任务无法完成

**影响**：
- 发布日期推迟，用户期待落空
- 项目成本增加（人力、硬件）
- 竞品可能抢先发布类似产品

**缓释措施**：
1. **Hook 引擎性能问题**：实现 VFS 缓存（Week 1）；若损耗仍 >15%，提供"性能模式"选项（降低 Hook 粒度）
2. **应用捕获失败率高**：优先捕获 10 款常用应用，逐步扩展；提供用户自定义规则（白名单/黑名单）
3. **Launchpad UI 开发进度延迟**：使用 Qt 6 快速原型开发；优先级排序，先做核心功能（应用管理、启动、商店）
4. **代码签名证书申请延迟**：Week 1 就开始申请（提前启动）；若延迟，先发布未签名版本（提示用户）
5. **团队成员离职或请假**：文档齐全，代码注释清晰；交叉培训（Cross-training）
