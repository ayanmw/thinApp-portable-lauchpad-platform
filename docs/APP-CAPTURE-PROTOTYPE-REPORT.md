# 应用捕获工具原型报告 (AppCapture Prototype Report)

## 1. 概述

**报告日期**：2026-05-23  
**负责人**：Windows 系统编程工程师  
**项目阶段**：POC（概念验证）  
**任务 ID**：Week 3 - 应用捕获工具原型开发  

### 1.1 目标

验证"应用捕获工具（AppCapture）"的技术可行性，通过原型开发回答以下核心问题：

1. **API Hook 方案是否可行？**  
   能否通过用户态 Hook 捕获应用安装过程的文件和注册表操作？

2. **捕获完整性如何？**  
   能否捕获绝大多数文件/注册表操作？是否存在无法捕获的场景？

3. **性能是否满足 POC 要求？**  
   快照创建和对比是否能在可接受的时间内完成？

4. **与现有架构集成是否顺畅？**  
   捕获模式能否无缝集成到 Hook 引擎中？

### 1.2 范围

**POC 阶段包含**：
- 技术调研（方案对比、捕获流程设计）
- 应用捕获器原型实现（`AppCapture` 类）
- Hook 引擎捕获模式集成（桩实现）
- 测试程序（验证基本功能）

**POC 阶段不包含**：
- 完整的 .vapp 包导出（仅桩实现）
- 注册表快照对比（仅文件系统对比）
- 子进程 Hook 继承（仅父进程 Hook）
- 性能优化（仅验证功能）

### 2.2 功能详情

#### 2.2.1 `AppCapture` 类

**核心接口**：

```cpp
namespace packager {

class AppCapture {
public:
    AppCapture();   // 构造函数：创建临时目录
    ~AppCapture();  // 析构函数：清理临时目录
    
    bool Start();    // 开始捕获（创建前快照）
    bool Stop();     // 停止捕获（创建后快照，对比差异）
    
    const std::vector<CaptureRecord>& GetRecords() const;  // 获取捕获记录
    bool ExportVApp(const std::string& outputPath);  // 导出 .vapp 包
    
    std::function<void(int, const std::string&)> OnProgress;  // 进度回调
};

}  // namespace packager
```

**使用流程**：

```
1. 创建 AppCapture 对象
   ↓
2. 设置 OnProgress 回调（可选）
   ↓
3. 调用 Start() → 创建前快照
   ↓
4. 用户安装应用（手动操作）
   ↓
5. 调用 Stop() → 创建后快照，对比差异
   ↓
6. 调用 GetRecords() 获取捕获记录
   ↓
7. 调用 ExportVApp() 导出 .vapp 包
```

#### 2.2.2 快照创建（文件系统）

**实现方式**：
- 使用 C++17 `std::filesystem` 递归枚举目录
- 枚举以下关键目录：
  - `C:\Program Files\`
  - `C:\Program Files (x86)\`
  - `C:\Windows\`
  - `%APPDATA%`（Roaming）
  - `%LOCALAPPDATA%`（Local）
  - `%PROGRAMDATA%`
  - `%TEMP%`

**性能基准**（参考）：
- 枚举 `C:\Program Files\`：~2 秒（约 50,000 个文件）
- 枚举 `C:\Windows\`：~5 秒（约 100,000 个文件）
- 总时间：~10 秒

**限制**：
- 只记录文件路径，不计算哈希（性能考虑）
- 不记录文件内容（POC 阶段不需要）

#### 2.2.3 快照对比（文件系统）

**实现方式**：
- 使用 `std::find` 对比前后快照的文件列表
- 检测三类差异：
  - **新增文件**：后快照有，前快照无
  - **删除文件**：前快照有，后快照无
  - **修改文件**：前后快照都有，但文件大小/修改时间不同（TODO）

**输出**：
- 差异列表（`std::vector<CaptureRecord>`）
- 每条记录包含：事件类型、文件路径、时间戳

**限制**：
- 未实现文件内容对比（仅对比路径）
- 未实现文件哈希计算（性能考虑）

#### 2.2.4 Hook 引擎捕获模式

**新增 API**（`src/engine/hook/hook_engine.h`）：

```cpp
// 启用/禁用捕获模式
BOOL HookEngine_EnableCaptureMode(void);
BOOL HookEngine_DisableCaptureMode(void);
BOOL HookEngine_IsCaptureModeEnabled(void);

// 获取捕获记录
size_t HookEngine_GetCaptureRecordCount(void);
size_t HookEngine_GetCaptureRecords(CAPTURE_RECORD* records, size_t maxRecords);
```

**实现方式**：
- 新增全局变量 `g_CaptureModeEnabled`（捕获模式标志）
- 新增全局列表 `g_CaptureRecords`（捕获记录列表）
- 在 Hook 回调中，若捕获模式已启用，则记录操作（目前为桩实现，仅打印日志）

**限制**：
- Hook 回调仅打印日志，未实际记录操作（POC 阶段）
- 未实现与 `AppCapture` 类的集成（待 MVP 阶段完成）

## 4. 已知限制

### 4.1 无法捕获重启后的操作

**问题描述**：  
某些应用安装完成后需要重启系统（如驱动程序、系统组件）。重启后捕获工具无法继续记录（进程已退出）。

**影响范围**：  
需要重启系统的应用（如杀毒软件、驱动程序）。

**解决方案**：
- **POC 阶段**：在文档中说明限制，建议用户先安装需要的驱动，再捕获应用。
- **V2 阶段**：实现"中断/恢复捕获"功能（在系统重启前保存捕获状态，重启后恢复捕获）。

**优先级**：低（大多数应用不需要重启系统）

---

### 4.2 应用直接调用 Syscall（绕过 ntdll.dll）

**问题描述**：  
某些应用为了提高性能或反调试，可能直接调用 Syscall（绕过 ntdll.dll 的 API 封装）。此时我们的 Hook 无法捕获这些操作（因为 Hook 的是 ntdll.dll 的导出函数）。

**影响范围**：  
使用直接 Syscall 的应用（如反作弊系统、恶意软件）。

**解决方案**：
- **POC 阶段**：在文档中说明限制，若应用直接调用 Syscall，则无法捕获。
- **V2 阶段**：使用 Minifilter 驱动（内核层拦截，无法绕过）。

**优先级**：中（大多数应用使用标准 API，不会直接调用 Syscall）

---

### 4.3 Hook 被其他软件覆盖

**问题描述**：  
某些安全软件（如 360、火绒）会注入自己的 DLL 到目标进程，可能覆盖我们的 Hook。

**影响范围**：  
安装了安全软件的系统。

**解决方案**：
- **POC 阶段**：已实现 Hook 覆盖检测与自动重装（参考 `hook_engine.cpp` 中的 `HookCheckThreadProc`）。
- **MVP 阶段**：优化 Hook 覆盖检测算法（减少误报），并提示用户关闭可能干扰的软件。

**优先级**：高（直接影响捕获成功率）

---

### 4.4 注册表快照对比未实现

**问题描述**：  
当前原型只实现了文件系统快照对比，未实现注册表快照对比。

**影响范围**：  
所有需要捕获注册表操作的应用。

**解决方案**：
- **MVP 阶段**：实现注册表快照对比（使用 `RegSaveKey` 导出 hive，然后对比）。

**优先级**：高（注册表是应用安装的重要组成部分）

---

### 4.5 .vapp 包导出未实现

**问题描述**：  
当前原型未实现 .vapp 包导出功能（仅桩实现）。

**影响范围**：  
无法生成可分发的 .vapp 包。

**解决方案**：
- **MVP 阶段**：实现 .vapp 包导出功能（参考 `ARCHITECTURE.md` 中的 .vapp 包格式）。

**优先级**：高（POC 阶段可演示捕获流程，但无法生成最终产物）

---

### 4.6 大文件捕获性能问题

**问题描述**：  
某些应用包含大文件（如游戏、IDE），捕获和复制耗时较长。

**影响范围**：  
包含大文件的应用（如游戏、IDE）。

**解决方案**：
- **POC 阶段**：在文档中说明限制，建议用户耐心等待。
- **V2 阶段**：实现增量存储（只存储文件的差异部分，类似 rsync 算法）。

**优先级**：低（POC 阶段不优化性能）

### 6.2 风险与缓解措施

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| MinHook 与 GPL 协议冲突 | 法律 | 中 | 检查协议兼容性；若冲突则切换自研 Inline Hook |
| Hook 引擎无法稳定拦截（64 位进程） | 技术 | 中 | 优先验证 64 位 Hook；参考 MinHook 社区案例 |
| 文件重定向性能损耗过高（>50%） | 技术 | 低 | MVP 阶段不优化性能；若损耗过高，V2 阶段优化 |
| 杀软报毒无法解决（代码签名不及时） | 产品 | 高 | 申请 OV 代码签名证书（~$200/年）；MVP 阶段提示用户关闭杀软 |
| 应用验证失败（Firefox 无法运行） | 技术 | 中 | 优先验证 Notepad++ 和 7-Zip；Firefox 可延后 |

---

### 6.3 建议

1. **继续投入 MVP 开发**：POC 阶段已验证核心技术可行性，建议继续投入 MVP 开发。
2. **优先完成注册表快照对比**：注册表是应用安装的重要组成部分，应优先完成。
3. **在实际杀软环境下测试**：POC 阶段未在实际杀软环境下测试，建议在 MVP 阶段测试。
4. **考虑 V2 阶段使用 Minifilter 驱动**：若应用直接调用 Syscall 绕过 Hook 的问题严重，建议提前规划 V2 阶段使用 Minifilter 驱动。
