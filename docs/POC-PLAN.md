# POC 验证计划 (POC Plan)

## 1. POC 目标

验证"用户态 API Hook + 文件系统/注册表虚拟化"技术路线的可行性，产出可演示的原型，为 MVP 决策提供工程依据。

**核心问题**：
1. Hook 引擎能否稳定拦截目标 API？
2. 文件系统/注册表重定向是否完整可用？
3. 杀软误报是否可控？
4. 典型应用能否在虚拟环境中正常运行？

### 4.1 测试详细步骤

#### V1：Hook 引擎稳定性测试
```
1. 安装 Hook（5 个 API）
2. 调用原函数（透传测试）
3. 卸载 Hook
4. 重复步骤 1-3 共 100 次（压力测试）
5. 检查内存泄漏（任务管理器 / Process Explorer）
6. 连续运行 24 小时，每小时触发一次 Hook API 调用
7. 检查崩溃日志
```
**通过标准**：步骤 4 无错误；步骤 5 内存增长 < 5MB；步骤 6 无崩溃

#### V2：文件系统重定向测试
```
1. 启动 Hook 引擎
2. 应用调用 CreateFile("C:\Windows\Temp\test.txt", GENERIC_WRITE)
3. 验证文件出现在 {AppDir}\VFS\C\Windows\Temp\test.txt
4. 应用调用 WriteFile(hFile, "hello")
5. 验证 {AppDir}\VFS\C\Windows\Temp\test.txt 内容为 "hello"
6. 应用调用 ReadFile("C:\Windows\Temp\test.txt")
7. 验证读取内容来自 {AppDir}\VFS\C\Windows\Temp\test.txt（若沙箱有）
8. 若沙箱无，验证读取内容来自真实 C:\Windows\Temp\test.txt（透传）
9. 应用调用 DeleteFile("C:\Windows\Temp\test.txt")
10. 验证沙箱文件被删除；真实文件不受影响
```
**通过标准**：步骤 3/5/7/10 全部符合预期

#### V3：注册表重定向测试
```
1. 启动 Hook 引擎
2. 应用调用 RegCreateKey(HKCU\Software\TestKey)
3. 验证虚拟 hive 中出现 HKCU\Software\TestKey
4. 应用调用 RegSetValueEx(HKCU\Software\TestKey, "TestValue", "hello")
5. 验证虚拟 hive 中 TestValue = "hello"
6. 应用调用 RegQueryValueEx(HKCU\Software\TestKey, "TestValue")
7. 验证读取内容来自虚拟 hive（若虚拟 hive 有）
8. 若虚拟 hive 无，验证读取内容来自真实注册表（透传）
```
**通过标准**：步骤 3/5/7 全部符合预期

#### V5：杀软兼容性测试
```
1. 在干净 Windows 环境中安装 Windows Defender（默认启用）
2. 运行 Hook 引擎（安装 Hook）
3. 记录 Defender 是否报毒、报毒类型（恶意/可疑/无需操作）
4. 重复步骤 2-3 共 10 次
5. 在干净 Windows 环境中安装火绒安全软件
6. 重复步骤 2-4（用火绒替代 Defender）
```
**通过标准**：Defender + 火绒均不报毒，或仅首次提示"允许/阻止"且可选择"允许"

## 6. Go / No-Go 决策标准

### 6.1 Go（继续 MVP）
**条件**：V1 **且** V2 **且** V3 **且** V5 全部通过

**含义**：技术路线可行，核心功能可验证，无明显阻塞项，可以进入 MVP 阶段

### 6.2 No-Go（暂停投入，技术路线调整）
**条件**：V1 **或** V2 **或** V5 任一失败

**含义**：核心技术无法验证，需要重新评估技术路线（如切换到内核驱动方案）

**失败定义**：
- V1 失败：Hook 引擎无法稳定工作（崩溃/内存泄漏严重）
- V2 失败：文件系统重定向不完整（大量文件泄漏到真实系统）
- V5 失败：杀软持续报毒且无法申请白名单（项目无法分发）

### 6.3 Conditional Go（有条件继续）
**条件**：V1+V2+V3+V5 通过，但 V4/V6/V7/V8 部分失败

**含义**：核心技术可行，但部分功能需要 MVP 阶段修复

**处理方式**：记录失败项，纳入 MVP 任务清单
