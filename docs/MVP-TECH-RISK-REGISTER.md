# MVP 技术风险登记册 (MVP Tech Risk Register)

> 本文档记录 AI ThinApp Portable Launchpad Platform 项目 MVP 阶段的所有已识别技术风险。
> 基于 POC 风险登记册（RISK-REGISTER.md）扩展和深化。
> 更新频率：每周评审一次，新风险随时添加。

---

## 风险矩阵定义

| 影响 \ 概率 | 低 (L) | 中 (M) | 高 (H) |
|------------|--------|--------|--------|
| **致命 (F)** | 中 | 高 | **极高** |
| **高 (H)** | 低 | 中 | 高 |
| **中 (M)** | 低 | 低 | 中 |
| **低 (L)** | 极低 | 低 | 低 |

**风险等级** = 影响 × 概率
- 🔴 极高：必须立即处理，项目可能因此终止
- 🔴 高：需在下一阶段开始前处理
- 🟡 中：需在版本内处理
- 🟢 低：可记录观察，不主动处理
- ⚪ 极低：仅记录

## 与 POC 风险的关系

本文档是 POC 风险登记册（RISK-REGISTER.md）的延续和深化。MVP 阶段的技术风险比 POC 更高，因为需要实现完整功能，而非原型。

### POC → MVP 风险演进

| POC 风险 ID | MVP 风险 ID | 演进说明 |
|-------------|-------------|----------|
| R-TECH-02 | R-MVP-06 | POC 阶段仅需检测 Hook 覆盖，MVP 需要自动恢复和多层防护 |
| R-TECH-04 | R-MVP-01 | POC 阶段仅验证功能，MVP 需要优化性能（缓存、算法） |
| R-TECH-05 | R-MVP-01 | POC 阶段接受性能损耗，MVP 需要降低到可接受范围（<15%） |
| R-TECH-07 | R-MVP-07 | POC 阶段仅验证基础继承，MVP 需要支持 5 层继承和复杂场景 |
| R-PROJ-02 | R-MVP-03 | POC 阶段提示用户关闭杀软，MVP 需要彻底解决误报问题 |
