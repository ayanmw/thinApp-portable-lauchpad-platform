## 1. 当前架构分析

### 1.1 Hook安装流程
当前Hook引擎基于MinHook库实现，安装流程如下：
1. 调用`MH_Initialize()`初始化MinHook
2. 遍历需要Hook的API列表（CreateFileW、RegCreateKeyW、RegSetValueExW、CreateProcessW）
3. 对每个API：
   - 调用`GetProcAddress()`获取目标API地址
   - 调用`MH_CreateHook()`创建钩子，保存原API指针
   - 调用`MH_EnableHook()`启用钩子
4. 启动Hook覆盖检测线程

**存在问题**：
- 无错误恢复机制：任一API Hook失败则整个安装过程返回失败
- 无回滚机制：部分Hook成功后失败，无法回滚已安装的钩子
- 日志输出性能开销大：生产环境应移除`printf`调用

### 1.2 文件重定向实现
**当前状态**：POC阶段仅实现钩子框架，未实现实际文件重定向逻辑。

**预期实现逻辑**：
- 在`HookedCreateFileW`中，将目标文件路径重定向到VFS目录
- 例如：将`C:\Program Files\App\data.txt`重定向到`{AppDir}\VFS\C\Program Files\App\data.txt`

**性能瓶颈**：
- 每次文件操作都需进行全路径匹配和字符串替换，开销大
- 无缓存机制，重复路径需重复计算

### 1.3 注册表重定向实现
**当前状态**：POC阶段仅实现钩子框架，未实现实际注册表重定向逻辑。

**预期实现逻辑**：
- 在`HookedRegCreateKeyW`和`HookedRegSetValueExW`中，将注册表路径重定向到虚拟注册表Hive
- 例如：将`HKEY_CURRENT_USER\Software\App`重定向到`{AppDir}\VReg\HKCU_Software_App.hive`

**性能瓶颈**：
- 每次注册表操作都需进行路径匹配，无缓存
- Hive文件IO开销大，未实现内存缓存

### 1.4 子进程继承实现
**当前状态**：POC阶段仅实现钩子框架，未实现子进程继承逻辑。

**预期实现逻辑**：
- 在`HookedCreateProcessW`中，为新创建的子进程注入Hook DLL
- 支持多层子进程继承（当前MVP目标为5层）

**性能瓶颈**：
- 远程线程注入开销大，未优化
- WOW64场景（32位/64位进程交叉）未处理

### 1.5 性能瓶颈总结
| 瓶颈类型 | 具体表现 | 影响程度 |
|----------|----------|----------|
| 字符串操作 | 路径匹配、字符串替换未优化 | 高 |
| 文件IO | VFS路径映射无缓存，重复计算 | 高 |
| 注册表IO | Hive文件读写无缓存 | 高 |
| Hook覆盖检测 | 无实际检测逻辑，定时检测频率不合理 | 中 |
| 子进程注入 | 注入逻辑未优化，不支持多层继承 | 中 |

## 2. VFS缓存设计

### 2.1 缓存策略
- **缓存策略**：LRU（最近最少使用） + TTL（生存时间）
- **LRU容量**：最大缓存1000个路径映射
- **TTL**：60秒（超时自动失效）
- **大小限制**：最大100MB（防止内存占用过高）

### 2.2 缓存数据结构
采用**哈希表 + 双向链表**结构：
- **哈希表**：键为原始路径，值为双向链表节点指针，支持O(1)查找
- **双向链表**：按访问时间排序，最近访问的节点在链表头部，最久未访问的在尾部
- **LRU淘汰**：当缓存满时，淘汰链表尾部节点

### 2.3 缓存失效策略
- **文件修改检测**：定期检查缓存文件的修改时间，若变化则失效
- **TTL到期**：缓存项创建/更新超过60秒自动失效
- **手动失效**：应用更新时主动清理缓存

### 2.4 性能目标
- 缓存命中率 ≥ 80%
- 重定向延迟 < 1ms（命中缓存时）
- 内存占用 < 50MB（1000个缓存项）

## 3. 路径重定向算法优化设计

### 3.1 当前算法分析
若采用朴素字符串替换或正则表达式匹配：
- 时间复杂度：O(n*m)（n为路径长度，m为规则数量）
- 性能问题：规则数量多时（如100+），匹配速度极慢

### 3.2 优化方案
#### 方案1：Trie树（前缀树）
- **数据结构**：每个节点代表一个路径字符，根节点代表空路径
- **查询复杂度**：O(k)（k为路径长度），与规则数量无关
- **适用场景**：路径前缀匹配（如`C:\Program Files\`重定向到`{AppDir}\VFS\C\Program Files\`）

#### 方案2：前缀匹配 + 缓存
- **前缀匹配**：将常用路径前缀（如`C:\Windows\`、`C:\Program Files\`）预先计算重定向结果
- **缓存**：将计算结果缓存到VFS缓存中，避免重复计算

#### 方案3：混合方案（推荐）
- 使用Trie树存储所有路径重定向规则
- 将Trie树查询结果缓存到VFS缓存中
- 兼顾查询速度和内存占用

### 3.3 性能目标
- 路径重定向延迟 < 0.1ms（命中缓存时）
- Trie树内存占用 < 10MB（1000条规则）
- 支持1000+路径重定向规则

## 4. Hook覆盖检测增强设计

### 4.1 当前实现分析
- 当前实现：`HookCheckThreadProc`线程每5秒（5次`Sleep(1000)`）检测一次
- 存在问题：
  - 无实际检测逻辑（代码中存在`TODO`注释）
  - 检测频率不合理：5秒间隔可能漏掉中间被卸载的钩子
  - 无自动重装机制

### 4.2 增强方案
#### 检测点增强
- **定时检测**：将检测间隔缩短为1秒（平衡性能和实时性）
- **事件驱动检测**：
  - 进程创建时检测（在`HookedCreateProcessW`中调用检测逻辑）
  - DLL加载时检测（在`DllMain`中调用检测逻辑）
- **多检测点冗余**：同时启用定时检测和事件驱动检测，提高可靠性

#### 自动重装策略
- **指数退避重试**：
  - 第1次重试：等待1秒
  - 第2次重试：等待2秒
  - 第3次重试：等待4秒
  - 第4次重试：等待8秒
  - 最多重试5次，若仍失败则记录错误并停止重试
- **重装流程**：
  1. 调用`MH_DisableHook()`禁用被覆盖的钩子
  2. 调用`MH_EnableHook()`重新启用钩子
  3. 若失败则触发指数退避重试

### 4.3 性能目标
- Hook覆盖检测延迟 < 1秒（事件驱动检测）
- 自动重装成功率 ≥ 95%
- 检测线程CPU占用 < 1%

## 5. 子进程继承完善设计

### 5.1 当前实现分析
- 当前状态：未实现子进程继承逻辑
- 存在问题：
  - 不支持多层子进程继承（MVP目标为5层）
  - 未处理WOW64场景（32位进程创建64位进程或反之）
  - 注入失败无异常处理，可能导致父进程崩溃

### 5.2 完善方案
#### 支持5层子进程继承
- **实现逻辑**：
  1. 在`HookedCreateProcessW`中，为新创建的子进程注入Hook DLL
  2. 子进程的Hook DLL加载时，继续为其创建的孙进程注入Hook DLL
  3. 递归支持最多5层继承
- **技术实现**：
  - 使用`CreateRemoteThreadEx()`替代`CreateRemoteThread()`，支持更多参数（如WOW64场景）
  - 在注入的DLL中，通过导出函数通知父进程注入结果

#### 处理WOW64场景
- **检测方法**：调用`IsWow64Process()`检测进程位数
- **注入策略**：
  - 若父进程为32位、子进程为64位：使用64位Hook DLL注入
  - 若父进程为64位、子进程为32位：使用32位Hook DLL注入
- **兼容性处理**：同时提供32位和64位Hook DLL

#### 异常处理
- **注入失败处理**：
  1. 捕获`CreateRemoteThreadEx()`失败错误
  2. 记录错误日志（包含进程ID、错误码）
  3. 返回错误码给父进程，不崩溃父进程
- **继承深度限制**：超过5层后不再注入，避免无限递归

### 5.3 性能目标
- 子进程注入成功率 ≥ 99%（非WOW64场景）
- 子进程注入成功率 ≥ 95%（WOW64场景）
- 注入延迟 < 100ms/进程
- 支持5层子进程继承

## 9. 风险与依赖

### 9.1 依赖
- 依赖`docs/RISK-MVP-01-HOOK-PERFORMANCE.md`中定义的性能风险缓释措施
- 依赖MinHook库（版本≥1.3.3）

### 9.2 风险
| 风险描述 | 影响程度 | 缓释措施 |
|----------|----------|----------|
| VFS缓存锁管理复杂，可能导致死锁 | 高 | 使用读写锁（SRWLOCK），读操作不阻塞，写操作阻塞 |
| Trie树内存占用过高 | 中 | 限制Trie树节点数量，定期清理不常用节点 |
| WOW64场景注入失败率高 | 中 | 增加WOW64场景测试用例，优化注入逻辑 |
| Hook覆盖检测误报率高 | 低 | 增加检测次数阈值，避免单次检测失败就触发重装 |

### 9.3  blocker记录
若存在技术难点（如高性能并发锁），立即记录到`docs/MVP-BLOCKERS.md`并上报PM。
