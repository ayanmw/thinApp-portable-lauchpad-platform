# Launchpad UI 技术选型文档

## 1. 概述

本文档对比多种桌面 UI 开发技术方案，为 AI ThinApp Portable Launchpad Platform 项目的 Launchpad UI 组件选择最合适的技术栈。

**评估目标**：
- POC 阶段快速验证交互原型（V9）
- 将 HTML 原型转化为桌面应用
- 保证后续 MVP 和 V2 阶段的可扩展性

---

## 2. 技术方案对比

### 方案 A：Win32 API + C++

| 项目 | 说明 |
|------|------|
| **技术栈** | 原生 C++，直接调用 Win32 API |
| **优点** | 1. 性能最好，无额外运行时开销<br>2. 无外部依赖，部署简单<br>3. 完全控制 UI 渲染和消息循环 |
| **缺点** | 1. 开发速度慢，代码量大<br>2. UI 美观度差，需自绘控件<br>3. 跨平台困难（仅 Windows）<br>4. 现代 UI 效果（动画、阴影）实现复杂 |
| **推荐？** | ❌ 否（MVP 不推荐） |
| **适用场景** | 对性能和体积有极致要求的场景 |

---

### 方案 B：Qt 6 + C++ ⭐ 推荐

| 项目 | 说明 |
|------|------|
| **技术栈** | Qt 6 框架（C++17） |
| **优点** | 1. UI 美观，支持现代效果（QSS 样式、动画）<br>2. 跨平台（Windows/Linux/macOS）<br>3. 与项目技术栈一致（C++17）<br>4. 开源（LGPL 协议，动态链接不要求开源）<br>5. 社区成熟，文档丰富<br>6. 支持 Qt Widgets（传统桌面 UI）和 Qt Quick（现代 QML UI） |
| **缺点** | 1. 静态链接需要商业授权（GPL 可用动态链接）<br>2. 学习曲线较陡（信号槽机制、元对象系统）<br>3. 动态链接需附带 Qt DLL（约 40-60 MB） |
| **推荐？** | ✅ **是（MVP 推荐）** |
| **适用场景** | 跨平台桌面应用，需要现代 UI 效果 |

**Qt 6 授权说明**：
- LGPL v3 协议：动态链接 Qt 库，不需要开源自己的代码
- 商业授权：需要静态链接或闭源分发时购买（约 $500/开发者/月）

---

### 方案 C：C# + WPF

| 项目 | 说明 |
|------|------|
| **技术栈** | .NET 6+，WPF（XAML） |
| **优点** | 1. UI 美观，支持现代效果（动画、渐变、阴影）<br>2. 开发速度快（XAML 声明式 UI）<br>3. 数据绑定强大（MVVM 模式）<br>4. Visual Studio 设计器支持拖拽 |
| **缺点** | 1. 需要 .NET 6+ Runtime（约 100 MB）<br>2. 仅支持 Windows（跨平台需 .NET 8+ 和 MAUI，不成熟）<br>3. 社区开源生态不如 Qt |
| **推荐？** | ⚠️ 备选 |
| **适用场景** | Windows 专用应用，团队有 .NET 经验 |

---

### 方案 D：C# + WinForms

| 项目 | 说明 |
|------|------|
| **技术栈** | .NET 6+，WinForms |
| **优点** | 1. 开发最快，学习曲线低<br>2. Visual Studio 拖拽设计器成熟<br>3. 兼容性好（支持旧版 .NET） |
| **缺点** | 1. UI 美观度一般（原生 Windows 控件风格）<br>2. 需要 .NET 6+ Runtime<br>3. 仅支持 Windows<br>4. 现代 UI 效果（圆角、阴影）实现困难 |
| **推荐？** | ⚠️ 备选 |
| **适用场景** | 内部工具，对 UI 要求不高的场景 |

---

### 方案 E：WebView2 + HTML/CSS/JS

| 项目 | 说明 |
|------|------|
| **技术栈** | Edge WebView2 嵌入本地 HTML |
| **优点** | 1. 完全复用现有 HTML 原型（index.html）<br>2. Web 技术生态丰富（Tailwind CSS、Font Awesome）<br>3. 前后端分离，便于迭代<br>4. UI 效果最佳（CSS 动画、渐变） |
| **缺点** | 1. 需要 Edge WebView2 Runtime（约 150 MB）<br>2. 原生功能调用需通过 WebView2 桥接（JS ↔ C++）<br>3. 性能不如原生 UI（渲染开销）<br>4. 打包体积大（需附带 HTML/CSS/JS 资源） |
| **推荐？** | ❌ POC 不推荐（有趣，但复杂度高） |
| **适用场景** | 已有 Web 前端，需快速封装为桌面应用的场景 |

---

## 3. 推荐方案：Qt 6 + C++

### 3.1 推荐理由

1. **与项目技术栈一致**：项目核心为 C++17，使用 Qt 6 无需引入新语言（C#）。
2. **UI 美观**：Qt Widgets 支持 QSS（类似 CSS），可实现圆角、渐变、阴影等现代效果。
3. **开源协议友好**：LGPL v3 协议允许动态链接，不需要开源自己的代码。
4. **跨平台能力**：未来可扩展到 Linux 和 macOS（POC 阶段仅 Windows）。
5. **社区成熟**：Qt 官方文档详细，Stack Overflow 上问题解答丰富。

### 3.2 技术细节

- **UI 框架**：Qt Widgets（传统桌面 UI，适合本项目的卡片布局）
- **构建系统**：CMake（与项目一致）
- **编译器**：MSVC 2019+ 或 MinGW 11.2+
- **Qt 版本**：Qt 6.5 LTS（长期支持版）

### 3.3 依赖项

- Qt 6.5 LTS（动态链接，附带 Qt6Core.dll、Qt6Widgets.dll 等）
- CMake 3.16+
- C++17 编译器

---

## 4. 备选方案：C# + WPF

### 4.1 适用场景

若团队无 Qt 经验，但有 .NET 经验，可选此方案。

### 4.2 技术细节

- **UI 框架**：WPF（XAML）
- **构建系统**：MSBuild（Visual Studio 项目）
- **运行时**：.NET 6+ Runtime

### 4.3 优缺点

- **优点**：开发速度快，XAML 声明式 UI 易于迭代。
- **缺点**：仅支持 Windows，需要 .NET Runtime。

---

## 5. POC 阶段技术选型决策

| 决策项 | 结论 |
|--------|------|
| **主方案** | Qt 6 + C++（Qt Widgets） |
| **备选方案** | C# + WPF（若 Qt 6 安装/配置遇到问题） |
| **不推荐方案** | Win32 API、WebView2（POC 阶段复杂度高） |
| **后续优化** | MVP 阶段完善 UI（QSS 样式），V2 阶段集成开始菜单替换 |

---

## 6. 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Qt 6 安装失败 | POC 无法继续 | 备选方案：改用 C# + WPF |
| LGPL 协议理解错误 | 法律问题 | 动态链接 Qt 库，不静态链接 |
| Qt 学习曲线陡 | 开发速度慢 | 参考官方示例，优先实现核心功能 |

---

## 7. 结论

**推荐方案**：**Qt 6 + C++（Qt Widgets）**

**理由**：
1. 与项目技术栈一致（C++17）
2. UI 美观，支持现代效果
3. 开源协议友好（LGPL v3）
4. 社区成熟，文档丰富
