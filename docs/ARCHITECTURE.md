## 2. 核心模块设计

### 2.1 Hook 引擎模块 (`src/engine/hook/`)

#### 2.1.1 技术选型
- **Hook 框架**：MinHook (https://github.com/TsudaKageyu/minhook)
  - 轻量级，支持 x86/x64
  - 社区活跃，MIT 协议（与 GPL 兼容需确认）
  - 备选：Microsoft Detours (MS EULA，商业友好但 GPL 不兼容)
  - **决策**：MVP 用 MinHook；若协议冲突则自研 Inline Hook

#### 2.1.2 目标 Hook API

| API | 模块 | 用途 |
|-----|------|------|
| `NtCreateFile` | ntdll.dll | 文件创建/打开拦截 |
| `NtWriteFile` | ntdll.dll | 文件写入拦截 |
| `NtReadFile` | ntdll.dll | 文件读取（CoW 需判断） |
| `NtOpenFile` | ntdll.dll | 文件打开拦截 |
| `NtCreateKey` | ntdll.dll | 注册表键创建拦截 |
| `NtSetValueKey` | ntdll.dll | 注册表值设置拦截 |
| `NtOpenKey` | ntdll.dll | 注册表键打开拦截 |
| `CreateProcessInternalW` | kernel32.dll | 进程创建拦截（子进程继承） |
| `ShellExecuteExW` | shell32.dll | Shell 执行拦截 |
| `LoadLibraryW` | kernel32.dll | DLL 加载拦截（注入 Hook DLL） |

#### 2.1.3 Hook 安装流程
```
1. 应用程序启动
2. 注入 Hook DLL (远程线程注入 / AppInit_DLLs /  SetWindowsHookEx)
3. DllMain 中调用 MinHook 安装目标 API Hook
4. 保存原函数地址（用于转发/CoW）
5. Hook 函数执行：
   a. 检查目标路径是否需要重定向
   b. 若是 → 修改路径参数 → 调用原函数（重定向后路径）
   c. 若否 → 直接调用原函数（透传）
6. 异常处理：Hook 失败 → 降级策略（放行 / 拦截并提示）
```

#### 2.1.4 异常处理与恢复
- 定期检查 Hook 是否被覆盖（其他软件注入覆盖了我们的 Hook）
- 自动重装 Hook（每 5 秒检查一次，可配置）
- Hook 失败超过阈值 → 通知用户，建议重启应用

### 2.2 虚拟文件系统模块 (`src/engine/vfs/`)

#### 2.2.1 路径映射规则

| 真实路径（应用视角） | 虚拟路径（实际存储） |
|---------------------|---------------------|
| `C:\Windows\*` | `{AppDir}\VFS\C\Windows\*` |
| `C:\Program Files\*` | `{AppDir}\VFS\C\Program Files\*` |
| `C:\Program Files (x86)\*` | `{AppDir}\VFS\C\Program Files (x86)\*` |
| `%APPDATA%\*` (Roaming) | `{AppDir}\VFS\AppData\Roaming\*` |
| `%LOCALAPPDATA%\*` | `{AppDir}\VFS\AppData\Local\*` |
| `%PROGRAMDATA%\*` | `{AppDir}\VFS\ProgramData\*` |
| `%TEMP%\*` / `%TMP%\*` | `{AppDir}\VFS\Temp\*` |
| `%SYSTEMROOT%\*` | `{AppDir}\VFS\Windows\*` (同 C:\Windows) |
| `\\?\C:\*` (长路径) | `{AppDir}\VFS\C\*` (保留长路径支持) |

#### 2.2.2 Copy-on-Write (CoW) 逻辑
```
读取文件 X:
  1. 检查 {AppDir}\VFS\...X... 是否存在
  2. 若存在 → 读取虚拟文件（应用已修改过的版本）
  3. 若不存在 → 读取真实文件 X（透传）

写入文件 X:
  1. 若 X 在拦截路径列表内 → 重定向到 {AppDir}\VFS\...X...
  2. 若 X 不在拦截路径列表内 → 透传（允许写入真实系统，可配置）
```

#### 2.2.3 文件锁处理
- 应用持有文件锁时，虚拟文件也需模拟锁行为
- 使用 `LockFile` / `UnlockFile` 拦截
- 虚拟文件锁仅对当前应用沙箱有效（不影响真实系统）

### 2.3 虚拟注册表模块 (`src/engine/vreg/`)

#### 2.3.1 虚拟 Hive 格式
- 基于 Windows 注册表 hive 格式的简化实现
- 存储位置：`{AppDir}\VFS\registry.hive`
- 格式：二进制，兼容 Windows 注册表 API (`RegLoadMounterKey`)
- 备选：JSON 格式（人类可读，但性能较差）→ MVP 用 JSON，V2 迁移到二进制

#### 2.3.2 注册表路径映射

| 真实路径（应用视角） | 虚拟路径（实际存储） |
|---------------------|---------------------|
| `HKLM\Software\*` | `registry.hive\HKLM\Software\*` |
| `HKCU\Software\*` | `registry.hive\HKCU\Software\*` |
| `HKCR\*` (文件关联) | `registry.hive\HKCR\*` |
| `HKU\*` (用户配置) | `registry.hive\HKU\*` |

#### 2.3.3 CoW 逻辑（同 VFS）
- 读取时先查虚拟 hive，不存在则透传真实注册表
- 写入时重定向到虚拟 hive

### 2.4 进程隔离模块 (`src/engine/process/`)

#### 2.4.1 子进程 Hook 继承
```
父进程启动子进程:
  1. 拦截 CreateProcessInternalW
  2. 修改 STARTUPINFO (若需要)
  3. 创建挂起子进程 (CREATE_SUSPENDED)
  4. 在子进程中注入 Hook DLL (远程线程)
  5. 恢复子进程执行
  6. 子进程 DllMain 中自动安装 Hook → 继承完成
```

#### 2.4.2 多进程架构支持
- Chrome/Edge：多进程模型（Browser 进程 + Renderer 进程 + GPU 进程）
  - 每个进程独立 Hook，互不影响
  - 进程间通信（IPC）可选隔离（V2）
- Firefox：多进程（Parent 进程 + Content 进程）
- Office：单进程 + 多线程（Hook 需支持多线程安全）

#### 2.4.3 COM 对象虚拟化 (V2)
- 拦截 `CoCreateInstance` / `CoGetClassObject`
- 虚拟 COM 对象注册（无需写入真实注册表）
- 支持 In-Process 和 Out-of-Process COM 组件

### 2.5 应用打包模块 (`src/packager/`)

#### 2.5.1 应用捕获流程
```
1. 拍摄前快照:
   - 扫描文件系统（C:\Windows, C:\Program Files, %APPDATA%, 等）
   - 导出注册表 HKLM\Software 和 HKCU\Software
   - 记录当前运行进程列表

2. 用户安装应用（手动或静默）

3. 拍摄后快照:
   - 再次扫描文件系统（差异分析）
   - 再次导出注册表（差异分析）
   - 记录新增/修改/删除的文件和注册表项

4. 生成差异包:
   - 文件系统差异 → {AppDir}\VFS\...
   - 注册表差异 → {AppDir}\VFS\registry.hive
   - 元数据 → manifest.json

5. 打包为 .vapp:
   - 压缩 VFS 目录 + registry.hive + manifest.json
   - 添加文件头（magic: "VAPP"，版本号，偏移量）
```

#### 2.5.2 .vapp 包格式

```
偏移量      内容
=========  ==========================================
0x0000     Magic Number: "VAPP" (4 bytes)
0x0004     版本号 (uint16 major, uint16 minor)
0x0008     文件头大小 (uint32)
0x000C     文件数据块偏移量 (uint32)
0x0010     文件数据块大小 (uint32)
0x0014     注册表 hive 块偏移量 (uint32)
0x0018     注册表 hive 块大小 (uint32)
0x001C     元数据 JSON 偏移量 (uint32)
0x0020     元数据 JSON 大小 (uint32)
0x0024     ... 保留 ...
0x0100     文件头结束，文件数据块开始
...         文件数据 (压缩，zlib/zip)
...         注册表 hive 块 (压缩)
...         元数据 JSON (未压缩，UTF-8)
```

#### 2.5.3 包编辑器功能
- 查看：文件树浏览，注册表树浏览，元数据查看
- 编辑：添加/删除/修改文件，添加/删除/修改注册表项，编辑元数据
- 合并：将多个 .vapp 包合并为一个（解决依赖）
- 导出：导出为目录（便于调试），导出为单 .exe（便于分发）

- **GUI 框架**：Electron (跨平台潜力) 或 Win32 (原生性能)

## 4. 数据流向

### 4.1 应用启动流程
```
用户点击"启动"按钮
        │
        ▼
Launchpad 检查应用状态
        │
        ├── 未安装 → 提示用户去商店安装或添加本地应用
        │
        └── 已安装 → 准备启动环境
                │
                ▼
        设置环境变量 (AppDir = 应用目录)
                │
                ▼
        注入 Hook DLL 到目标进程
                │
                ▼
        启动应用 (CreateProcess)
                │
                ▼
        应用运行，所有文件/注册表操作被 Hook 拦截
                │
                ▼
        Hook 引擎重定向路径 → 读写 {AppDir}\VFS\...
                │
                ▼
        应用正常运行，真实系统无污染
```

### 4.2 文件写入拦截流程
```
应用调用 NtCreateFile("C:\Windows\temp\app.log", GENERIC_WRITE)
        │
        ▼
Hook 引擎拦截，检查路径
        │
        ├── 路径在拦截列表内?
        │       │
        │       ├── 是 → 修改路径为 {AppDir}\VFS\C\Windows\temp\app.log
        │       │       调用原 NtCreateFile (重定向后路径)
        │       │       返回虚拟文件句柄
        │       │
        │       └── 否 → 直接调用原 NtCreateFile (透传)
        │
        └── 返回文件句柄给应用
                │
                ▼
应用调用 NtWriteFile(hFile, "log content")
        │
        ▼
Hook 引擎不拦截写入（已获得虚拟文件句柄，写入自动到虚拟文件）
        │
        ▼
写入成功
```

## 5. 技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| MinHook 与 GPL 协议冲突 | 法律 | 检查协议兼容性；若冲突则自研 Inline Hook |
| Hook 被其他软件覆盖 | 功能失效 | 定期检查 + 自动重装 Hook |
| 应用使用直接 Syscall (绕过 ntdll) | 拦截失败 | V2 增加内核 Minifilter 驱动 |
| 杀软误报 | 用户体验差 | 代码签名 + 开源透明 + 申请白名单 |
| 性能损耗过高 | 用户体验差 | 热路径优化 + 缓存 + 异步 IO |
| 兼容性（应用 X 无法运行） | 功能受限 | 兼容性规则库 + 用户自定义规则 |
